## Key Decisions
_Non-obvious choices made — why, not just what._
- **`request.DATA` duplicate keys resolve later-source-wins — never a list-merge.** `RequestDataParser._set_nested_value` assigns unconditionally; precedence is query < form < JSON body, whole-value replace (no deep-merge of nested dicts). Multi-value stays list-shaped only *within* one source (`?tag=a&tag=b`, `tags[]=`, JSON lists — they arrive in a single call). Do NOT reinstate the collision merge branch: a cross-source dup (`?group=518` + body `{"group":518}` — routine web-mojo traffic) used to become `['518', 518]`, and the dispatcher's `int(request.DATA.group)` (`mojo/decorators/http.py`) runs BEFORE `dispatch_error_handler` wraps the view, so the `TypeError` escaped as a bare Django 500 logged only to `django.request` (invisible to logit). Dispatcher coercion now catches `(TypeError, ValueError)` → JSON 400 (incident detail bounded `repr(...)[:200]`); `requires_perms`/`requires_group_perms` group fallback treats an unusable `group` as no-group-context → fail-closed 403. `touch()`/`is_group_allowed` deliberately stay INSIDE the dispatcher's try — narrowing it reroutes their ValueError to a bare 500 (the real fix, wrapping the whole pre-dispatch block, belongs with the `dispatcher-group-param-is-active-asymmetry` inbox item). Regression: `tests/test_middleware/request_data_merge.py`. (ITEM-024)
- OTP/verification flows are **retry-safe**: read → compare → consume the secret/session ONLY on success (never `getdel`-before-compare). Brute force is bounded by the per-IP rate limit + TTL, NOT per-session attempt counters — consistent across `_verify_otp`, `verify_phone_verify_code`, and `phone_register.verify_code`. Do not "harden" by deleting on a wrong attempt; that burns the session and dead-ends the happy path. (ITEM-005) — Where consume-FIRST is required (to prevent duplicate users / double-firing `USER_REGISTERED_HANDLER`), instead **restore** the consumed token if the post-consume work fails: `phone_register.restore()`, called in `on_register`'s `except` path around the handler-firing atomic block (scoped so post-handler failures keep it consumed). Don't "simplify" that try/except away. (ITEM-008)
- **Geofence exemption/config invariants (ITEM-017):** decision priority is bypass → IP allowlist → rules; an allowlisted request still **shadow-evaluates** the rules so `geofence_exempt` evidence carries `would_block` — don't "optimize" to a fast-path return. Evidence emission lives in the **decorator** (cache hits still emit; `geo/check`/simulate stay silent by design). `lookup_failed` decisions are **never cached** (posture is scope-sensitive via `GEOFENCE_FAIL_CLOSED_SCOPES` and scope isn't in the cache key). Decision-cache invalidation + `geofence_config` events hook the **model layer** (`Setting.save/delete` on `GEOFENCE_KEYS`, `Group.on_rest_saved`, `GeoLocatedIP.whitelist/unwhitelist`) so REST, programmatic, and shell writes all stay coherent. `whitelist_active` (expiry-aware) is the only correct whitelist read — an expired whitelist neither exempts geofence nor suppresses firewall blocks; `whitelisted_until` never federates. — Evidence metrics (ITEM-020): base slugs `geofence:blocks`/`geofence:exempt` dual-write to `account=group-<pk>` sourced from **`request.group` only** (the group the engine evaluated with; never `user.org` — blocks are mostly pre-auth and org didn't participate in the decision); country/region suffixed slugs stay **global-only** (groups × geography key cross-product; months/years metric keys never expire). Reads were already gated (`metrics/rest/helpers.py` group-account permission check) — no new read surface. — Strict posture (ITEM-021): `GEOFENCE_STRICT_POSTURE` is one bundled opt-in switch (fail-closed **OR**s with the flag+scope map, deny-private, deny-no-rules `no_rules_strict`); the no-rules deny lives in `_evaluate()` AFTER the allowlist step — moving it earlier locks allowlisted office IPs out. Per-group override `Group.metadata["geofence_strict"]` is a flat sibling key (never nest inside `metadata["geofence"]` — that's still the raw rule dict) and changing it requires the **global** `manage_geofence`/`security` perm (owner ruling 2026-07-09: tenant admins must not opt out of a platform posture; detected by DB compare since JSONField merges skip changed_fields) + emits `geofence_config` evidence. `tor_exits`/`blocklist_de` IPSet rows are `is_cache_only`: enable action 400s and `sync()` hard no-ops — never let them reach the kernel firewall. Test hygiene: never persist a global strict=true Setting row in tests (parallel unheadered requests all 403 `no_rules_strict`) — use the header or a test-owned group override. — Member plane (ITEM-022): `GET /api/geo/policy` is the ONE member-scoped geofence read, gated `requires_perms("view_security","security")` — deliberately `Event.VIEW_PERMS`, so a single member grant lights up policy + own-group event feed; `view_geofence`/`manage_geofence` stay global/config-plane-only and do NOT open it. Response is built solely from `request.group` (dispatcher-resolved; `group_uuid` = active groups only; unknown uuid 403s identically to unauthorized — no existence oracle) with a deliberately narrow payload — never `enforced_endpoints`/allowlist/cache_ttl/fail-closed scopes/config provenance; the anti-leak key list is asserted in `tests/test_geofence/member_visibility.py`, so don't add fields there without updating the contract deliberately. Members READ strict posture; writes stay global (ITEM-021 ruling). Parent-group grants cover child groups (`check_parents`). Events needed no new mechanics (RestMeta group fallback already scopes; groupless events stay member-invisible — attribution caveat documented both tracks). — Settings validation (ITEM-023): `Setting.register_validator(key, func, global_only=True)` is the ONE write-validation mechanism — enforced in `on_rest_pre_save` AND `save()` (Setting.set/shell covered; queryset `.update()`/`bulk_create` are the only accepted bypasses), validator signature `func(key, parsed)` raising ValueError, value must be valid JSON, registered keys reject group-scoped rows AND `is_secret` (secret used to skip validation + mask the value — don't reintroduce that early-return). All 8 `GEOFENCE_*` keys are registered and the same `GEOFENCE_KEYS` tuple drives decision-cache invalidation — add a geofence key → register a validator AND extend the tuple. Read side: `settings.get(kind=...)` degrades garbage to the DECLARED default + `logit.warning("settings", ...)` that logs the value's TYPE only (resolve returns decrypted secrets — never log raw values there); unrecognized bool strings are no longer truthy; never reintroduce `objict.from_json(..., ignore_errors=True)` in coercion (it masks parse errors as an empty objict).
- **Group membership/permission lookups must be ApiKey-safe at the choke point.** `request.user` / `active_user` can be a **bare `ApiKey`** under `Authorization: apikey <token>`, not a `User`. `Group.get_member_for_user` guards with `if not hasattr(user, "is_request_user"): return None` — the framework's canonical "is this a real request `User`?" idiom (also `group.py:216`, `rest.py` 241/264/465) — so it never runs the `User`-typed `members.filter(user=…)` that raises `Must be "User" instance.` for an ApiKey. The fix lives at the **single choke point** because `user_has_permission` was already guarded but 3 direct callers bypassed it (`check_view_permission`, `on_group_me_member`, `can_change_permission`). A non-`User` identity → `None`/deny everywhere (fail-closed); grant/deny for keys flows through `ApiKey.has_permission` (group-scoped; `sys.*` always denied, unchanged). Don't add a `filter(user=request.user)` / member lookup without this guard, and don't special-case ApiKey per-caller — guard the choke point. (ITEM-016)
- **Client IP (`request.ip` / WS `remote_ip`) comes from `X-Real-IP`, never `X-Forwarded-For`.** Both `get_remote_ip` (HTTP) and the realtime WS resolver (`apps/realtime/handler.py` `resolve_remote_ip`/`get_remote_ip`) read the proxy-authoritative `X-Real-IP` (the universal `asgi.inc` sets it to `$remote_addr`, overwriting any client value), fall back to `REMOTE_ADDR`/transport-peer, and normalize via the shared public `normalize_ip` (`mojo/helpers/request.py`; IP:port / bracketed / IPv4-mapped IPv6). The leftmost `X-Forwarded-For` and RFC 7239 `Forwarded` are client-spoofable, and `scope["client"]` is uvicorn-XFF-derived — do NOT reinstate reading them or prefer them over `X-Real-IP`. Holds for direct + load-balanced deploys; every deployment MUST set `X-Real-IP`. (ITEM-009/ITEM-010)

## Watch List
_Fragile areas, known debt, things to tread carefully._
- **An ApiKey identity is denied by default on GROUPLESS `uses_model_security` models.** `MojoModel._evaluate_permission`'s api_key branch (`mojo/models/rest.py`) fails closed when a key's request isn't confined to a group (no `group` FK on the model, or a null-group instance) — else a group key self-claiming `manage_users` reads every tenant's `User` rows (proven end-to-end). Per-model opt-out `RestMeta.ALLOW_API_KEY_GLOBAL` (default False; none set it). Models that OVERRIDE `check_view_permission`/`on_rest_handle_list` bypass Branch B and need their own key guard — `Group.check_view_permission` uses `api_key.is_group_allowed(self)` to confine a key to its own group tree. `ApiKey.permissions` assignment is gated by `can_change_permission`/`set_permissions` + `APIKEY_PERMS_PROTECTION` (kind=dict, default {}); `create_for_group` assigns directly (ungated, trusted). Machine access to global data = dedicated `allow_api_keys` endpoint or a service-account User, never a group key. (ITEM-019)
- **Global-effect endpoints MUST use `@md.requires_global_perms(...)`, never `@md.requires_perms(...)`.** `requires_perms` falls back to the caller's group/member permission for the client-supplied `group` param (`REQUIRES_PERMS_IS_GROUP` default True); GroupMember perms accept arbitrary keys any group admin can assign → member-scoped grant + own group id = cross-tenant escalation on any endpoint whose effect isn't confined to `request.group`. `requires_global_perms` (`mojo/decorators/auth.py`) checks global `User.permissions`/superuser only and **rejects ApiKey identities** (via `hasattr(user,"is_request_user")` — User has it, ApiKey doesn't) unless `allow_api_keys=True` (only the geoip/sync federation receiver uses that). ITEM-018 switched ~45 endpoints (jobs, aws, account admin, metrics, incident, assistant, geofence). Deliberately LEFT on `requires_perms` because genuinely group-scoped: `group/webhook_secret`, `group/member*`, `phonehub sms/send` (apikey-federated, group-attributed), aws `on_rest_request` delegators, assistant memory, `device/location`. Regression: `tests/test_global_perms/`. (ITEM-018)
- **`dev_server.conf` (testit dev-server host/port) has THREE readers that must stay in sync** — two Python (`testit/runner.py:get_host`, `testit/helpers.py:_read_dev_server_conf`) + one **bash** (`bin/asgi_local:_read_conf`, the one that actually binds uvicorn, easy to miss). All resolve the effective file the same way: prefer `var/dev_server.conf` (gitignored, per-machine) else committed `config/dev_server.conf` — **whole-file replace, no per-key merge**. Python routes through `paths.resolve_conf()`; bash duplicates it inline (can't import the helper). Change one → change all three, or the test client and server disagree on host/port. Tests: `tests/test_helpers/test_runner_config.py` (tmp dirs only — never write live `var/*.conf`; uvicorn's `--reload-include '*.conf'` would reload mid-suite). (ITEM-014)
- **Geofence test hygiene (ITEM-017):** NEVER prime a cached geofence DENY under the shared `(127.0.0.1, no-group)` decision-cache key, and never DB-allowlist/whitelist `127.0.0.1` — parallel modules' unheadered auth requests run with the default cache TTL (300) and real DB rules, so they get served the poison (this silently broke `test_email` mid-suite). Group-scope the primed key (`geo/check?group_uuid=` on a test-owned group). Also: direct `POST /api/auth/login` calls trip the login **muid** rate tier (10/300s per `_muid` session cookie) — clear it the way `testit client.login()` does (`clear_rate_limits(key="login", muid=cookie)`), not just the ip bucket. Also: `Group.objects.create()` leaves `uuid=None` — it's lazily assigned; call `grp.get_uuid()` before driving `group_uuid` params, else the dispatcher's group resolution (http.py) silently no-ops and `request.group` stays None (ITEM-020 hit this).
- **Overriding a Django setting in an _in-process_ testit test → direct `setattr(django.conf.settings, …)` with try/finally restore, NOT `th.server_settings`.** `th.server_settings` reloads the separate _server_ process (for `opts.client` tests); it has zero effect on a setting read via `settings.get_static()`/`get()` in code the test calls **directly** (same process). `override_settings` is banned by the testing rules. Pattern: save `getattr(dj, name, sentinel)`, set, then in `finally` `delattr` (if originally absent) else restore — see the `_override_setting` contextmanager in `tests/test_jobs/test_signed_webhook.py` + `tests/test_account/test_webhook_signer.py`. (ITEM-015)
