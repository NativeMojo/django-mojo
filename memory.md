## Key Decisions
_Non-obvious choices made — why, not just what._
- **`request.DATA` duplicate keys resolve later-source-wins — never a list-merge.** `RequestDataParser._set_nested_value` assigns unconditionally; precedence is query < form < JSON body, whole-value replace (no deep-merge of nested dicts). Multi-value stays list-shaped only *within* one source (`?tag=a&tag=b`, `tags[]=`, JSON lists — they arrive in a single call). Do NOT reinstate the collision merge branch: a cross-source dup (`?group=518` + body `{"group":518}` — routine web-mojo traffic) used to become `['518', 518]`, and the dispatcher's `int(request.DATA.group)` (`mojo/decorators/http.py`) runs BEFORE `dispatch_error_handler` wraps the view, so the `TypeError` escaped as a bare Django 500 logged only to `django.request` (invisible to logit). Dispatcher coercion now catches `(TypeError, ValueError)` → JSON 400 (incident detail bounded `repr(...)[:200]`); `requires_perms`/`requires_group_perms` group fallback treats an unusable `group` as no-group-context → fail-closed 403. `touch()`/`is_group_allowed` deliberately stay INSIDE the dispatcher's try — narrowing it reroutes their ValueError to a bare 500. Regression: `tests/test_middleware/request_data_merge.py`. (DM-024) — **Numeric `group=` resolves ACTIVE groups only** via `Group.get_active(pk)` — the ONE resolver for client-supplied numeric group ids (dispatcher + both auth-decorator fallbacks); inactive == nonexistent (silent None, no touch, no oracle — same contract as the `group_uuid` branch). Don't reintroduce an unfiltered lookup there, and don't distinguish inactive from nonexistent in any response. KNOWN deliberate leftovers (inbox items filed 2026-07-10): RestMeta LIST member fallback (`get_groups_with_permission` ignores `Group.is_active`), `ApiKey.validate_token` group context (deactivated group's keys keep working — docs say "deactivate the key"), `group/<pk>/member` oracle+touch. Regression: `tests/test_middleware/group_param_is_active.py`. (DM-025)
- **OAuth providers on the hosted auth pages are tuple-driven — `LOGIN_METHODS`/`REGISTRATION_METHODS` in `services/auth_config.py` are the single source of truth** for config validation, `DEFAULT_AUTH_CONFIG` (defaults = ALL tuple members, so tuple membership ⇒ default-ON for every group without an explicit methods list), the begin gate (`rest/oauth.py` `provider in LOGIN_METHODS` → `assert_login_method`; no-op without group context), and the new-user registration gate (`provider_name in REGISTRATION_METHODS`). DM-026 added `github` this way (owner ruling 2026-07-10: default-on full parity; opt-in rejected — it would 403 existing group-scoped github API begins). Adding a provider = tuple(s) + button in `_login_method_buttons.html`/`login.html`/`register.html` + `auth_base.html` (`ENABLE_<X>` flag, hide line, **section-hide guard** — miss it and an X-only config hides its own button row) + `startXLogin` wrapper in `mojo-auth.js` + `ALL_LOGIN_METHODS` in `tests/test_auth_config/auth_config.py`. Callback spinner lookup is the generic `$("btn-" + provider)` — don't reintroduce a per-provider ternary. No config-presence check gates button rendering (unset `GITHUB_CLIENT_ID` → dead button, mirrors google — deliberate, documented in auth_pages.md).
- **Geofence exemption/config invariants (DM-017):** decision priority is bypass → IP allowlist → rules; an allowlisted request still **shadow-evaluates** the rules so `geofence_exempt` evidence carries `would_block` — don't "optimize" to a fast-path return. Evidence emission lives at the **enforcement point** — the decorator wrapper pre-view, `enforcement.enforce()` for DM-043's post-credential checks (cache hits still emit; `geo/check`/simulate stay silent by design). `lookup_failed` decisions are **never cached** (posture is scope-sensitive via `GEOFENCE_FAIL_CLOSED_SCOPES` and scope isn't in the cache key). Decision-cache invalidation + `geofence_config` events hook the **model layer** (`Setting.save/delete` on `GEOFENCE_KEYS`, `Group.on_rest_saved`, `GeoLocatedIP.whitelist/unwhitelist`) so REST, programmatic, and shell writes all stay coherent. `whitelist_active` (expiry-aware) is the only correct whitelist read — an expired whitelist neither exempts geofence nor suppresses firewall blocks; `whitelisted_until` never federates. — Evidence metrics (DM-020): base slugs `geofence:blocks`/`geofence:exempt` dual-write to `account=group-<pk>` sourced from **`request.group` only** (the group the engine evaluated with; never `user.org` — blocks are mostly pre-auth and org didn't participate in the decision); country/region suffixed slugs stay **global-only** (groups × geography key cross-product; months/years metric keys never expire). Reads were already gated (`metrics/rest/helpers.py` group-account permission check) — no new read surface. — Strict posture (DM-021): `GEOFENCE_STRICT_POSTURE` is one bundled opt-in switch (fail-closed **OR**s with the flag+scope map, deny-private, deny-no-rules `no_rules_strict`); the no-rules deny lives in `_evaluate()` AFTER the allowlist step — moving it earlier locks allowlisted office IPs out. Per-group override `Group.metadata["geofence_strict"]` is a flat sibling key (never nest inside `metadata["geofence"]` — that's still the raw rule dict) and changing it requires the **global** `manage_geofence`/`security` perm (owner ruling 2026-07-09: tenant admins must not opt out of a platform posture; detected by DB compare since JSONField merges skip changed_fields) + emits `geofence_config` evidence. `tor_exits`/`blocklist_de` IPSet rows are `is_cache_only`: enable action 400s and `sync()` hard no-ops — never let them reach the kernel firewall. Test hygiene: never persist a global strict=true Setting row in tests (parallel unheadered requests all 403 `no_rules_strict`) — use the header or a test-owned group override. — Member plane (DM-022): `GET /api/geo/policy` is the ONE member-scoped geofence read, gated `requires_perms("view_security","security")` — deliberately `Event.VIEW_PERMS`, so a single member grant lights up policy + own-group event feed; `view_geofence`/`manage_geofence` stay global/config-plane-only and do NOT open it. Response is built solely from `request.group` (dispatcher-resolved; `group_uuid` = active groups only; unknown uuid 403s identically to unauthorized — no existence oracle) with a deliberately narrow payload — never `enforced_endpoints`/allowlist/cache_ttl/fail-closed scopes/config provenance; the anti-leak key list is asserted in `tests/test_geofence/member_visibility.py`, so don't add fields there without updating the contract deliberately. Members READ strict posture; writes stay global (DM-021 ruling). Parent-group grants cover child groups (`check_parents`). Events needed no new mechanics (RestMeta group fallback already scopes; groupless events stay member-invisible — attribution caveat documented both tracks). — Settings validation (DM-023): `Setting.register_validator(key, func, global_only=True)` is the ONE write-validation mechanism — enforced in `on_rest_pre_save` AND `save()` (Setting.set/shell covered; queryset `.update()`/`bulk_create` are the only accepted bypasses), validator signature `func(key, parsed)` raising ValueError, value must be valid JSON, registered keys reject group-scoped rows AND `is_secret` (secret used to skip validation + mask the value — don't reintroduce that early-return). All 8 `GEOFENCE_*` keys are registered and the same `GEOFENCE_KEYS` tuple drives decision-cache invalidation — add a geofence key → register a validator AND extend the tuple. Read side: `settings.get(kind=...)` degrades garbage to the DECLARED default + `logit.warning("settings", ...)` that logs the value's TYPE only (resolve returns decrypted secrets — never log raw values there); unrecognized bool strings are no longer truthy; never reintroduce `objict.from_json(..., ignore_errors=True)` in coercion (it masks parse errors as an empty objict).
- **Login-flow geofencing is POST-credential (DM-043) — five invariants.** (1) Identity-bearing auth endpoints carry `@md.requires_geofence(scope="auth", after_auth=True)` (registers in SECURITY_REGISTRY, annotated `after_auth: true` in `geo/rules` enforced_endpoints; wrapper does NOT block) — enforcement runs via the shared `services/geofence/enforcement.enforce(request, scope, user)` at the TOP of `jwt_login()` (before last_login/UserLoginEvent/USER_LOGIN_HANDLER — a blocked login has ZERO success side effects) plus the MFA branch of `on_user_login` (blocked users never receive an mfa_token). (2) `jwt_login` enforces for EVERY source by default; `GEOFENCE_EXEMPT_JWT_SOURCES = ("sessions_revoke", "email_change")` are the only exemptions (authed re-issues — a blocked-geo user must still revoke their own sessions). New login sources are geofenced automatically — don't add to the exempt tuple without an owner ruling. (3) Ordering contract: invalid credentials → the normal 401 regardless of geo (no enumeration/oracle change); valid credentials → the standard leak-scrubbed geofence 403 (accepted tradeoff, owner ruling 2026-07-17: valid-cred holders can distinguish 403 from 401). Token-proven actions (password reset / email verify / invite accept) APPLY their mutation, then withhold the session — accepted, documented. (4) `on_register` stays on the pre-view BLOCKING decorator (must block before account creation), and OAuth `/complete` enforces in-view right after the provider exchange, BEFORE `_find_or_create_user`, via lookup-only `_lookup_known_user` (security-review catch: jwt_login-only enforcement let a blocked-geo caller provision a new account + fire the registration webhook + persist provider tokens). Never move that call below the create path. (5) `evidence.report_block/report_exempt` take an explicit `user=` (attributed only when `is_authenticated`) because `request.user` stays anonymous on login endpoints even after in-view verification — the reporter can't derive it. KNOWN pre-existing bug found here (filed inbox `security-registry-decorator-clobbering`): ten auth.py decorators OVERWRITE SECURITY_REGISTRY entries, so enforced_endpoints under-reports whenever one sits above requires_geofence — enforcement itself is unaffected. Regressions: `tests/test_geofence/post_auth.py` (8), `tests/test_oauth/oauth.py` lookup-only test. (DM-043)
- **Group membership/permission lookups must be ApiKey-safe at the choke point.** `request.user` / `active_user` can be a **bare `ApiKey`** under `Authorization: apikey <token>`, not a `User`. `Group.get_member_for_user` guards with `if not hasattr(user, "is_request_user"): return None` — the framework's canonical "is this a real request `User`?" idiom (also `group.py:216`, `rest.py` 241/264/465) — so it never runs the `User`-typed `members.filter(user=…)` that raises `Must be "User" instance.` for an ApiKey. The fix lives at the **single choke point** because `user_has_permission` was already guarded but 3 direct callers bypassed it (`check_view_permission`, `on_group_me_member`, `can_change_permission`). A non-`User` identity → `None`/deny everywhere (fail-closed); grant/deny for keys flows through `ApiKey.has_permission` (group-scoped; `sys.*` always denied, unchanged). Don't add a `filter(user=request.user)` / member lookup without this guard, and don't special-case ApiKey per-caller — guard the choke point. (DM-016) — **`ANONYMOUS_USER` is the OTHER non-`User` identity that reaches permission checks** (`mojo/middleware/mojo.py`, set on every unauthenticated request). Do NOT reorder `Group.user_has_permission`'s `is_request_user` guard (group.py:216) to precede its `user.has_permission(perms)` call (group.py:214) — an ApiKey's grant flows through that call, so reordering denies valid keys. `POST /api/group/member/invite` also guards `request.group is None` → generic 403 (inactive == nonexistent, DM-025 contract). (DM-028) DM-029 then swept EVERY `has_permission`/`user_has_permission(request.user` site in `rest/**` + models and verified each is reached only via an auth-gated route (a `requires_*` decorator, the model-security 401 short-circuit at `rest.py:234-240` for RestMeta perms lacking `"all"`, or a self-gating `is_authenticated` check) — no other ungated site exists, so DM-028's sentinel is a redundant backstop not sole protection. Don't re-flag helpers like `_check_room_admin`/`can_change_permission`/`check_view_permission` from reading them in isolation ("no local gate" ≠ "anon-reachable" — check the callers' route decorators). (DM-029, closed no-op)
- **Client IP (`request.ip` / WS `remote_ip`) comes from `X-Real-IP`, never `X-Forwarded-For`.** Both `get_remote_ip` (HTTP) and the realtime WS resolver (`apps/realtime/handler.py` `resolve_remote_ip`/`get_remote_ip`) read the proxy-authoritative `X-Real-IP` (the universal `asgi.inc` sets it to `$remote_addr`, overwriting any client value), fall back to `REMOTE_ADDR`/transport-peer, and normalize via the shared public `normalize_ip` (`mojo/helpers/request.py`; IP:port / bracketed / IPv4-mapped IPv6). The leftmost `X-Forwarded-For` and RFC 7239 `Forwarded` are client-spoofable, and `scope["client"]` is uvicorn-XFF-derived — do NOT reinstate reading them or prefer them over `X-Real-IP`. Holds for direct + load-balanced deploys; every deployment MUST set `X-Real-IP`. (DM-009/DM-010)
- **The protected-JSON guard in `on_rest_update_jsonfield` covers EVERY write path — don't re-scope it to the merge branch.** The check is hoisted ABOVE the merge/replace split and duplicated in the outer non-dict branch: it fires on key **presence** in either direction (incoming dict carries `"protected"`, OR `should_replace`/non-dict overwrite would clobber an existing `protected` subtree), fail-closed even for a byte-identical no-op — same contract as merges always had. `__replace` is a **top-level-only** signal consumed by `field_value.pop`; `objict.merge_dicts` has NO nested-`__replace` handling (a nested one persists as a literal key — geofence.md used to document that nonexistent form; corrected). `JSON_REPLACE_FIELDS` (only `incident.Ticket.metadata`) rides the same guarded replace path. `_can_edit_protected_json` reads `is_superuser` via `getattr(..., False)` because `request.user` is a bare `ApiKey` under apikey auth (no such attr — the guarded merge path used to 500 for keys); ApiKey grant flows through `ApiKey.has_permission` (group-scoped). Audit (`meta:protected_changed`, unconditional): changed-keys unions in the EXISTING keys on replace and on a merge with explicit `"protected": null` (subtree wipe — merge_dicts drops null-valued keys), so removed keys are always named; that union was a security-review catch, don't simplify it back to incoming-only. Group's real perms are `["admin_compliance","admin_verify"]` (docs used to claim `manage_groups`). Regression: `tests/test_account/test_group_protected_metadata.py` (7 tests, incl. ApiKey 403-not-500 and null-wipe audit). (DM-030)

- **Deactivating a group suspends its API keys at REQUEST time — five coordinated gates, keys never mutated (reactivation restores instantly).** Owner ruling 2026-07-12: strip group context (403), NOT a 401 token reject — the federation path (`requires_global_perms(..., allow_api_keys=True)`, geoip `/sync`) authorizes on `has_permission` and ignores `request.group`, so an inactive-group fleet-peer key must still AUTHENTICATE. The five gates (a validate_token-only fix is insufficient; each closes a distinct bypass): (1) `validate_token` (api_key.py) sets `request.group` = active-group-or-None; (2) rest.py api_key branch re-checks `request.group.is_active` — the detail instance re-bind (rest.py ~290) repopulates the inactive group after the strip; (3) `ApiKey.get_groups` filters `is_active=True` — the RestMeta list fallback (`on_rest_handle_list` → `get_groups_with_permission`) re-enumerated the tenant's rows; (4) `ApiKey.is_group_allowed` requires the TARGET group active — `Group.check_view/edit_permission` run BEFORE gate (2) and a suspended tenant's key could otherwise read/write its own Group row incl. flipping `is_active` back (self-reversible suspension); (5) `requires_perms`/`requires_group_perms` trust a non-User identity's perm dict only with an ACTIVE `request.group` — the `has_permission` short-circuit otherwise passes custom endpoints (`sms/send`) with zero group consideration. Boundaries: real Users unaffected by (5); ~~per-group, NOT a cascade~~ **OVERTURNED by DM-048** (owner ruling 2026-07-17): all five gates now check `is_effectively_active()` — an active child under an inactive ancestor is dark too; `ALLOW_API_KEY_GLOBAL` is ignored+logged on any model with a group FK (only valid groupless). Don't re-simplify to fewer gates, and don't add an is_active check to `has_permission` itself (breaks federation). Sibling still open: User member-grant path (`User.get_groups` filters member.is_active, not `Group.is_active`) = inbox `member-perms-ignore-group-is-active`. Regression: `tests/test_global_perms/apikey_group_inactive.py` (8). (DM-037)

- **Machine-identity gates are STRUCTURAL as of DM-045 — three invariants.** (1) `mojo.helpers.request.is_request_user(request)` is the ONE person-vs-machine predicate (absence of the `is_request_user` marker = machine, fail-closed); both auth decorators (`_deny_machine_identity_without_active_group` — single extracted copy; its `not group.is_active` half is unreachable-today defense-in-depth, keep it) and `_evaluate_permission` key on it — never hand-roll `hasattr(user, "is_request_user")` in an authz decision again. - **A group is EFFECTIVELY active only if it AND every ancestor is active (DM-048, owner ruling 2026-07-17: "if a parent is disabled, all the children are also disabled").** `Group.is_effectively_active(max_depth=8)` is the SINGLE owner — never re-implement the ancestor walk or check bare `group.is_active` in an authz/resolution decision (a bare check misses a deactivated ancestor). Enforced DYNAMICALLY, never a flag cascade — cascading writes create a reactivation one-way door (can't tell cascade-disabled from deliberately-disabled children; same trap as the parked future/ apikey item); reactivating the parent must instantly restore individually-active descendants. Two choke points cover most surfaces for free: `Group.get_active` (dispatcher `group=`, decorator fallbacks, member endpoint) and the `get_member_for_user` TOP gate (`is_active=True` → `self.is_effectively_active()` before any membership query — one check proves every walked parent active; `is_active=False` = raw admin path, keep it raw). Also routed: `validate_token`/`is_group_allowed`/`get_groups`, both rest.py `api_key.group_inactive` gates, dispatcher `group_uuid`, registration/OAuth-state/geofence-preflight, bouncer `_resolve_group` + `auth_config.resolve_group_from_request` (UX-consistency), and `resolve_by_auth_domain` (chain verified per READ — the 24h Redis entry is keyed to the own-flag row and can't see ancestor flips; don't "fix" by caching effective state). `ApiKey.get_groups` derives the reachable subtree with ONE (id,parent_id) query + in-memory BFS from the key's group through own-flag-active nodes — do NOT reintroduce a per-descendant `is_effectively_active()` loop (N+1, security-review WARNING). Deliberate v1 boundaries: LIST surfaces (`LIST_DEFAULT_FILTERS`, `User.get_groups`, children listings) stay own-flag (SQL can't walk; disclosure-to-already-permitted, every gate denies); depth cap 8 (matches walk; >8 unverified, unsupported anyway); overturns the DM-037 per-group carve-out + flipped `test_apikey_active_child_still_reachable`. Regressions: oracle suite (+5), middleware (+1), apikey_group_inactive (+2). (DM-048)

## Watch List
_Fragile areas, known debt, things to tread carefully._
- **DM-042 abuse-hardening invariants:** (1) `check_api_throttle` is called at the TOP of `dispatcher()` (http.py) — before group resolution, deliberately; keyed by identity ONLY (user/apikey pk), anonymous = zero-cost skip (IP-keyed global limits = CGNAT trap). Accounting (INCR + bucket totals + top-talkers flush) runs even when `API_THROTTLE_ENABLED=False` — detection must not depend on 429 posture; don't "optimize" it behind the enabled flag. (2) EVERY 429 path (`_block`, `_throttle_block`, WS gates) emits metric+event only on FIRST engagement per window (SET NX) — never revert to per-rejection `report_event`: a failed request costing more than a served one is the doom-loop mechanism itself. (3) **Never rate-limit `incident/ossec/alert/batch`** — mojo-ossec treats 429 like any error and parks batches after 3 retries into a dir nothing re-reads (silent alert loss); its sender is already capped ~12 POSTs/min/host. - **An ApiKey identity is denied by default on GROUPLESS `uses_model_security` models.** `MojoModel._evaluate_permission`'s api_key branch (`mojo/models/rest.py`) fails closed when a key's request isn't confined to a group (no `group` FK on the model, or a null-group instance) — else a group key self-claiming `manage_users` reads every tenant's `User` rows (proven end-to-end). Per-model opt-out `RestMeta.ALLOW_API_KEY_GLOBAL` (default False; none set it). (DM-019) — **`_evaluate_permission` classifies read vs write by the RestMeta keys of the call** (write = keys contain `CREATE`/`SAVE`/`DELETE_PERMS`): a write SKIPS `check_view_permission` and gates on `check_edit_permission`; a read prefers the view hook (a model with only an edit hook — `User` — uses it for reads too).
