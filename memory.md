## Key Decisions
_Non-obvious choices made — why, not just what._
- **A malformed `Authorization` header is passthrough — never a 500, never a 401.** `AuthenticationMiddleware.process_request` guards `token.split()`: a bare scheme-less single token sets `request.auth_token = objict(prefix="raw", token=...)` and returns; empty / 3+-part headers just return; `request.bearer`/`request.user` stay unset in all these cases (fail-closed — protected endpoints still reject via the permission layer). `request.auth_token` is **informational only**, NOT an auth gate (the gate is `request.bearer`/`request.user`); its sole non-trivial reader `fresh_auth.token_auth_time` is `try/except`-guarded and bearer-gated. Do NOT 401 on a malformed header (breaks scheme-less webhooks, e.g. Coinflow) and do NOT treat `request.auth_token` presence as authentication. (ITEM-012)
- OTP/verification flows are **retry-safe**: read → compare → consume the secret/session ONLY on success (never `getdel`-before-compare). Brute force is bounded by the per-IP rate limit + TTL, NOT per-session attempt counters — consistent across `_verify_otp`, `verify_phone_verify_code`, and `phone_register.verify_code`. Do not "harden" by deleting on a wrong attempt; that burns the session and dead-ends the happy path. (ITEM-005) — Where consume-FIRST is required (to prevent duplicate users / double-firing `USER_REGISTERED_HANDLER`), instead **restore** the consumed token if the post-consume work fails: `phone_register.restore()`, called in `on_register`'s `except` path around the handler-firing atomic block (scoped so post-handler failures keep it consumed). Don't "simplify" that try/except away. (ITEM-008)
- **Geofence exemption/config invariants (ITEM-017):** decision priority is bypass → IP allowlist → rules; an allowlisted request still **shadow-evaluates** the rules so `geofence_exempt` evidence carries `would_block` — don't "optimize" to a fast-path return. Evidence emission lives in the **decorator** (cache hits still emit; `geo/check`/simulate stay silent by design). `lookup_failed` decisions are **never cached** (posture is scope-sensitive via `GEOFENCE_FAIL_CLOSED_SCOPES` and scope isn't in the cache key). Decision-cache invalidation + `geofence_config` events hook the **model layer** (`Setting.save/delete` on `GEOFENCE_KEYS`, `Group.on_rest_saved`, `GeoLocatedIP.whitelist/unwhitelist`) so REST, programmatic, and shell writes all stay coherent. `whitelist_active` (expiry-aware) is the only correct whitelist read — an expired whitelist neither exempts geofence nor suppresses firewall blocks; `whitelisted_until` never federates.
- **Group membership/permission lookups must be ApiKey-safe at the choke point.** `request.user` / `active_user` can be a **bare `ApiKey`** under `Authorization: apikey <token>`, not a `User`. `Group.get_member_for_user` guards with `if not hasattr(user, "is_request_user"): return None` — the framework's canonical "is this a real request `User`?" idiom (also `group.py:216`, `rest.py` 241/264/465) — so it never runs the `User`-typed `members.filter(user=…)` that raises `Must be "User" instance.` for an ApiKey. The fix lives at the **single choke point** because `user_has_permission` was already guarded but 3 direct callers bypassed it (`check_view_permission`, `on_group_me_member`, `can_change_permission`). A non-`User` identity → `None`/deny everywhere (fail-closed); grant/deny for keys flows through `ApiKey.has_permission` (group-scoped; `sys.*` always denied, unchanged). Don't add a `filter(user=request.user)` / member lookup without this guard, and don't special-case ApiKey per-caller — guard the choke point. (ITEM-016)
- **Client IP (`request.ip` / WS `remote_ip`) comes from `X-Real-IP`, never `X-Forwarded-For`.** Both `get_remote_ip` (HTTP) and the realtime WS resolver (`apps/realtime/handler.py` `resolve_remote_ip`/`get_remote_ip`) read the proxy-authoritative `X-Real-IP` (the universal `asgi.inc` sets it to `$remote_addr`, overwriting any client value), fall back to `REMOTE_ADDR`/transport-peer, and normalize via the shared public `normalize_ip` (`mojo/helpers/request.py`; IP:port / bracketed / IPv4-mapped IPv6). The leftmost `X-Forwarded-For` and RFC 7239 `Forwarded` are client-spoofable, and `scope["client"]` is uvicorn-XFF-derived — do NOT reinstate reading them or prefer them over `X-Real-IP`. Holds for direct + load-balanced deploys; every deployment MUST set `X-Real-IP`. (ITEM-009/ITEM-010)

## Watch List
_Fragile areas, known debt, things to tread carefully._
- **`@md.requires_perms` has a group-permission fallback (default on) — do NOT use it on endpoints whose effect isn't scoped to `request.group`.** A GroupMember-scoped grant (tenant-admin-assignable, arbitrary key strings, `member.py`) satisfies it via a client-supplied `group` param → cross-tenant escalation on global-effect endpoints. Geofence config plane uses `_requires_global_perms` (`rest/geofence.py`, global `User.permissions` only) + regression test. Other suspects (e.g. `jobs/rest/control.py`) unaudited — see `planning/inbox/requires-perms-group-fallback-audit.md`. (ITEM-017 security review)
- **`dev_server.conf` (testit dev-server host/port) has THREE readers that must stay in sync** — two Python (`testit/runner.py:get_host`, `testit/helpers.py:_read_dev_server_conf`) + one **bash** (`bin/asgi_local:_read_conf`, the one that actually binds uvicorn, easy to miss). All resolve the effective file the same way: prefer `var/dev_server.conf` (gitignored, per-machine) else committed `config/dev_server.conf` — **whole-file replace, no per-key merge**. Python routes through `paths.resolve_conf()`; bash duplicates it inline (can't import the helper). Change one → change all three, or the test client and server disagree on host/port. Tests: `tests/test_helpers/test_runner_config.py` (tmp dirs only — never write live `var/*.conf`; uvicorn's `--reload-include '*.conf'` would reload mid-suite). (ITEM-014)
- **Geofence test hygiene (ITEM-017):** NEVER prime a cached geofence DENY under the shared `(127.0.0.1, no-group)` decision-cache key, and never DB-allowlist/whitelist `127.0.0.1` — parallel modules' unheadered auth requests run with the default cache TTL (300) and real DB rules, so they get served the poison (this silently broke `test_email` mid-suite). Group-scope the primed key (`geo/check?group_uuid=` on a test-owned group). Also: direct `POST /api/auth/login` calls trip the login **muid** rate tier (10/300s per `_muid` session cookie) — clear it the way `testit client.login()` does (`clear_rate_limits(key="login", muid=cookie)`), not just the ip bucket.
- **Overriding a Django setting in an _in-process_ testit test → direct `setattr(django.conf.settings, …)` with try/finally restore, NOT `th.server_settings`.** `th.server_settings` reloads the separate _server_ process (for `opts.client` tests); it has zero effect on a setting read via `settings.get_static()`/`get()` in code the test calls **directly** (same process). `override_settings` is banned by the testing rules. Pattern: save `getattr(dj, name, sentinel)`, set, then in `finally` `delattr` (if originally absent) else restore — see the `_override_setting` contextmanager in `tests/test_jobs/test_signed_webhook.py` + `tests/test_account/test_webhook_signer.py`. (ITEM-015)
